# FocusGuard 로컬 번역·간편 배포 설계

## 목표

FocusGuard의 Google Apps Script 및 Google 계정 의존성을 제거한다. 인터넷 연결은 허용하며, 번역 요청은 앱에서 직접 전송하고 모든 사용자 데이터는 로컬에 저장한다. 기존 프로젝트 파일과 현재 진행 중인 테마 작업은 수정하지 않으며 별도 배포 프로젝트로 구성한다.

## 프로젝트 분리

- `src/FocusGuard.Local`에 별도 배포 프로젝트를 둔다.
- 기존 프로젝트의 UI, 리소스 및 최신 테마 파일을 연결해 사용한다.
- Apps Script 통신이 포함된 번역 창 코드와 로컬 데이터 저장에 필요한 코드만 배포 프로젝트 전용 구현으로 교체한다.
- 기존 `src/FocusGuard` 파일은 수정하지 않는다.
- 원본 테마 변경은 로컬 배포 프로젝트의 다음 빌드에 반영되도록 연결 구조를 사용한다.

## 번역 및 사전 검색

- 앱이 Google 번역 공개 엔드포인트에 직접 HTTPS 요청을 보낸다.
- Apps Script URL, Google 로그인, 사용자별 Apps Script 배포 과정은 모두 제거한다.
- 기존 영한 사전 결과 형식과 일반 번역 흐름을 가능한 범위에서 유지한다.
- 공개 엔드포인트가 비공식·무보증 인터페이스라는 점을 고려해 응답 파싱 실패와 네트워크 오류를 사용자에게 명확히 표시한다.
- 실패한 요청은 정상 검색 기록으로 저장하지 않는다.

## 로컬 데이터

- 데이터 루트는 `%LocalAppData%\FocusGuard\`로 통일한다.
- SQLite 데이터베이스와 사용자 설정을 해당 위치에 저장한다.
- 번역 기록에는 고유 ID, 시각, 유형, 입력문 및 결과를 저장한다.
- 기록 목록은 최신순으로 불러오며 새로고침과 개별 삭제를 지원한다.
- EXE 위치나 버전이 바뀌어도 기존 데이터가 유지되어야 한다.

## 배포 형식

최초 배포 폴더에는 다음 두 파일만 둔다.

- `FocusGuard-Setup.bat`
- `FocusGuard.exe`

EXE는 Windows x64용 self-contained 단일 파일로 게시해 별도 .NET 설치를 요구하지 않는다. BAT는 WebView2 Runtime 설치 여부를 확인하고, 없는 경우에만 공식 배포 경로에서 설치 프로그램을 내려받아 설치한다. 임시 설치 파일은 설치 완료 후 제거하며, 성공·실패 결과와 다음 행동을 사용자에게 알린다.

최종 출력 위치는 `dist/FocusGuard-local-win-x64`로 한다.

## 실행 및 오류 처리

- WebView2가 필요한 화면을 열 때 런타임이 없으면 설치 BAT 실행 안내를 표시한다.
- 네트워크 실패, HTTP 오류, 예상하지 못한 번역 응답 및 DB 오류를 구분해 표시한다.
- 오류가 나더라도 UI 버튼과 로딩 상태는 정상으로 복구되어야 한다.
- 로컬 DB 디렉터리는 첫 실행 때 자동 생성한다.

## 검증 및 출시 순서

1. 별도 프로젝트를 개발용 설정으로 빌드한다.
2. 프로그램을 직접 실행해 번역, 사전 검색, 기록 조회 및 삭제를 확인한다.
3. 사용자가 실행 중인 프로그램을 직접 확인한다.
4. 사용자 피드백이 있으면 수정한 뒤 다시 실행해 확인한다.
5. 사용자가 승인한 후에만 최종 단일 EXE와 설치 BAT를 게시한다.
6. 깨끗한 사용자 환경을 가정해 WebView2 감지·설치 흐름과 `%LocalAppData%` 데이터 유지를 검증한다.

## 완료 조건

- 기존 `src/FocusGuard` 파일에 이 기능을 위한 변경이 없다.
- 최신 테마가 별도 프로젝트 실행 화면에 반영된다.
- Apps Script나 Google 계정 없이 검색과 번역이 작동한다.
- 검색 기록의 생성, 조회 및 삭제가 로컬 SQLite에서 작동한다.
- 개발용 프로그램을 사용자가 확인하기 전에는 최종 배포물을 만들지 않는다.
- 최종 배포 폴더의 초기 파일이 BAT와 EXE 두 개뿐이다.
